# Hermes Web UI 安全热修设计

- 日期：2026-04-25
- 主题：公网单实例部署下的止血版安全热修
- 目标窗口：1-3 天
- 范围：代码修复 + 默认配置/部署建议
- 约束：
  - 优先修复真实可利用漏洞
  - 尽量保持现有客户端和部署方式兼容
  - 允许少量必要的破坏性变更来阻止高危误配置
  - 本文档只定义热修设计，不包含实现

## 1. 背景与目标

本次热修基于当前项目的安全审计结果，面向“公网可访问的单实例 self-hosted 管理面板”这一威胁模型。

热修目标不是一次性完成长期安全治理，而是在最短时间内收紧已经确认的高风险入口，避免以下情况继续存在：

- 已认证用户可读取宿主机任意文件
- profile 导入可利用文件名做路径穿越
- 大体积 multipart 请求可导致明显内存压力
- 启动和运行日志泄露完整认证 token
- `AUTH_DISABLED=1` 被误配后直接形成公网无认证管理面
- 登录面可被用户名枚举并进行无限制暴力尝试

## 2. 威胁模型

本次设计默认以下攻击者模型成立：

- 远程未认证攻击者可以访问公开暴露的 HTTP / WebSocket / Socket.IO 入口
- 攻击者可能能看到部署日志、应用日志或反向代理日志
- 攻击者可能已拿到一个静态 bearer token
- 运维可能沿用 README 中旧的“看启动日志拿 token”习惯
- 项目运行在单实例 Node 进程上，请求体处理若不受控，会直接影响服务稳定性

不作为本次热修前提的模型：

- 多租户细粒度权限隔离
- 完整 session/cookie 体系
- 分布式限流与集中式审计平台

## 3. 修复范围与优先级

### 3.1 P0：必须进入本次热修

1. 修掉任意文件读取
   - `/api/hermes/download` 不再允许读取任意宿主机绝对路径
   - 默认仅允许：
     - 当前 profile 根目录下的相对路径
     - 上传目录等明确白名单绝对路径

2. 修掉 profile 导入路径穿越
   - 导入归档时不再使用客户端提供的原始文件名参与临时落盘路径计算
   - 临时文件统一采用随机文件名

3. 修掉大请求造成的显著内存压力
   - `profiles/import` 和 `files/upload` 增加总请求体上限
   - 在读取请求流过程中累计大小，超限立即拒绝并停止继续处理

4. 启动和运行日志不再输出完整 token
   - 启动阶段不再把完整认证/API token 打到 `stdout`
   - `server.log` 及其他应用日志路径不得记录完整 token
   - 最多只允许输出脱敏值，或仅输出 `Auth enabled`

5. 收紧 `AUTH_DISABLED=1` 的危险语义
   - `AUTH_DISABLED=1` 不再默认等同于“关闭认证并继续对外提供完整 BFF + 终端能力”
   - 只有同时设置显式二次确认开关（例如 `ALLOW_INSECURE_NO_AUTH=1`）时，才允许进入不安全无认证模式
   - 否则进程应直接拒绝启动

6. 文档同步
   - 更新 `README.md` 与 `README_zh.md`
   - 去掉“从启动日志复制 token”这条旧路径
   - 改为推荐从服务所在机器本地读取 `~/.hermes-web-ui/.token`
   - `AUTH_TOKEN` 作为可选高级用法保留

### 3.2 P1：低风险可并入本次热修

1. 登录接口增加基础限流
   - 至少按 IP 做短窗限流
   - 阻止无限制密码猜测

2. `auth/status` 不再暴露真实用户名
   - 返回是否配置密码登录即可
   - 若为兼容前端保留字段，字段值应固定为 `null`

3. 高风险接口增加安全审计日志
   - 记录访问结果、来源、是否拒绝
   - 但不能记录 token、Authorization 头或完整敏感参数

### 3.3 明确不进入本次热修

- 不把静态 token 模型整体改为 session/cookie
- 不彻底移除 query token 支持
- 不重做权限模型或细粒度 RBAC
- 不做全面前端认证改造
- 不引入 Redis 等分布式基础设施作为热修前提

## 4. 兼容策略与允许的行为变化

本次热修整体采用“兼容优先”策略，但以下边界允许变化。

### 4.1 保持兼容的部分

- 继续保留 `Authorization: Bearer <token>` 认证方式
- 不改变现有前端登录主流程
- 不改变现有路由路径、主要 HTTP 方法和页面入口
- 相对路径文件下载继续按 profile 根目录解析
- terminal、group chat 等现有接入路径尽量不改协议和前端交互

### 4.2 允许的必要变化

1. 启动日志不再输出 token
   - 运维首次获取 token 的方式从“看日志”改为“本机读取 `~/.hermes-web-ui/.token`”

2. 任意宿主机绝对路径下载被禁止
   - 只有上传目录等白名单绝对路径继续允许

3. `AUTH_DISABLED=1` 变成受保护的不安全模式
   - 没有二次确认开关就拒绝启动
   - 这是本次热修允许的破坏性变更之一

4. 大文件上传/导入会更早失败
   - 超限请求直接返回 `413`
   - 不再等请求全部读完后再失败

### 4.3 本次暂不移除但降级处理的能力

- query token 本次不直接删除，避免带出较大兼容风险
- 但它会从推荐路径降级为兼容路径，后续版本应计划废弃

## 5. 代码修复方案

### 5.1 认证模式统一收口

目标：避免 HTTP、WebSocket、Socket.IO 各自判断，出现“某些入口已收紧、某些入口仍放开”的情况。

设计：

- 在认证服务层统一解析认证模式：
  - 正常认证模式
  - 显式不安全无认证模式
- 启动阶段前置校验：
  - 若 `AUTH_DISABLED=1` 且没有 `ALLOW_INSECURE_NO_AUTH=1`，直接启动失败
  - 若进入不安全模式，只输出高危警告，不输出 token
- terminal WebSocket 与 group chat Socket.IO 统一复用认证模式判断，不允许单独绕过

### 5.2 token 泄露面收口

目标：停止把静态 token 暴露给日志读取者、运维误操作和日志采集链路。

设计：

- 移除启动阶段 token 明文输出
- 清查应用日志中的 token 写入点
- 脱敏日志输出标准：
  - 优先不打印 token
  - 如必须打印标识，只允许掩码形式
- 文档中把初始 token 获取方式改为本机读取 `~/.hermes-web-ui/.token`

### 5.3 文件下载边界收口

目标：封住已认证后读取宿主机任意文件的能力，同时不破坏现有附件下载。

设计：

- `download` 接口采用显式边界判断：
  - 相对路径：继续解析到当前 profile 根目录
  - 绝对路径：
    - 若在上传目录白名单内，允许
    - 否则拒绝
- 对越权绝对路径统一返回 `403 permission_denied`
- 白名单逻辑应集中在文件服务层，而不是散落在各路由中

### 5.4 profile 导入路径穿越修复

目标：客户端文件名不再影响临时落盘目录边界。

设计：

- 导入归档时只读取扩展名和内容，不信任原始文件名路径部分
- 临时文件名改为服务端生成
- 仅允许归档扩展名白名单
- 临时目录外写入应在逻辑上不再可能发生

### 5.5 上传/导入请求体上限

目标：减少超大 multipart 请求对单实例 Node 进程的内存冲击。

设计：

- 为 `files/upload` 和 `profiles/import` 分别定义总请求体上限
- 在逐块读取请求流时累计大小
- 超限立即：
  - 返回 `413`
  - 停止继续解析
  - 不进入后续文件写入路径

本次热修要求“尽早拒绝”，但不强制要求立刻把现有 multipart 处理全面重写为完整流式解析器。

### 5.6 登录面止血

目标：降低公网暴露时的用户名枚举和暴力破解风险。

设计：

- `auth/status` 仅返回是否启用了密码登录，不返回真实用户名
- `auth/login` 增加基础 IP 限流
- 限流以“单进程可落地”为优先，可先采用进程内内存结构实现
- 限流失败返回明确但不泄露内部细节的错误

## 6. 默认配置与部署建议

### 6.1 生产默认建议

- 默认开启认证
- 默认不要使用 `AUTH_DISABLED=1`
- 若必须临时使用无认证模式，只限本机调试，并显式设置二次确认开关
- 推荐继续使用自动生成 token，首次从本机 `~/.hermes-web-ui/.token` 读取

### 6.2 网络暴露建议

- 不要把 8648 裸端口直接暴露在公网
- 应通过统一反向代理暴露
- 反向代理应统一覆盖：
  - `/api`
  - `/v1`
  - `/health`
  - `/upload`
  - `/webhook`
  - WebSocket / Socket.IO 升级路径

### 6.3 CORS 建议

- 当前默认 `*` 对公网管理面过宽
- 生产部署应显式设置 `CORS_ORIGINS`
- 若本次代码不改默认值，README 也必须明确标注生产必须配置

### 6.4 日志与反向代理建议

- 不在访问日志中记录 query token
- 不在错误日志中记录 Authorization 头
- 如果反向代理支持，建议对大请求做上限控制，避免只依赖 Node 侧拦截

## 7. 发布顺序与回滚策略

### 7.1 推荐发版顺序

1. 服务端安全修复
   - 认证模式校验
   - 下载边界收口
   - 导入路径修复
   - 请求体上限
   - 登录限流
   - 日志脱敏

2. 文档同步
   - token 获取方式
   - `AUTH_DISABLED` 新语义
   - 生产部署建议

3. 发版验证
   - 正常登录
   - 正常附件下载
   - profile 导入成功/超限失败
   - terminal 正常连接
   - `AUTH_DISABLED=1` 无 override 时启动失败

### 7.2 可回滚项

- 登录限流阈值
- 上传/导入大小上限数值
- 部分日志字段的详细程度

### 7.3 不应回滚的安全边界

- 启动日志打印完整 token
- 非白名单绝对路径下载
- `AUTH_DISABLED=1` 无二次确认直接放行
- profile 导入路径穿越

## 8. 测试与验收标准

### 8.1 自动化测试

1. 认证模式测试
   - `AUTH_DISABLED=1` 且无 override 时启动失败
   - 正常模式下受保护 HTTP / WS / Socket.IO 仍要求 token

2. 下载边界测试
   - 相对路径下载成功
   - 上传目录绝对路径下载成功
   - 非白名单绝对路径返回 `403`

3. 导入路径测试
   - 带路径分隔符或 `../` 的文件名不会逃逸临时目录
   - 合法归档仍可成功导入

4. 大请求限制测试
   - 超限请求返回 `413`
   - 超限后不进入后续写入路径

5. 登录面测试
   - `auth/status` 不再返回真实用户名
   - 多次失败登录会触发限流

6. 日志测试
   - 控制台和 `server.log` 中都不出现完整 token

### 8.2 手工回归

- 运维可按 README / README_zh 从本机读取 `~/.hermes-web-ui/.token` 完成首次登录
- 现有前端登录、聊天、附件下载、terminal 连接仍可正常使用
- 合法 profile 导入成功，异常文件名与超大文件被拒绝
- 生产配置下不开启 insecure override 也能正常认证运行

### 8.3 热修完成定义

以下条件全部满足，才视为热修完成：

- 宿主机任意文件读取被封住
- profile 导入路径穿越被封住
- 超大 multipart 请求不能轻易把进程拖入明显内存压力
- 控制台与应用日志不再泄露完整 token
- `AUTH_DISABLED=1` 不再能被误配为公网无认证管理面
- README / README_zh 已同步新的首次登录方式与生产安全提示

## 9. 非目标与后续工作

本次热修完成后，下一阶段可继续推进但不纳入本次交付：

- query token 全面下线
- token 轮换机制
- session/cookie 认证
- 更强的分布式限流
- 终端、group chat、文件操作的更细粒度权限控制
- `CORS_ORIGINS` 默认值进一步收紧

## 10. 结论

本设计采用“兼容优先、边界止血”的策略：优先收紧已经确认可利用的高风险点，尽量不改变现有客户端主流程；只有在阻止高危误配置时，才允许引入必要的破坏性变化。

如果严格按本设计执行，本次热修应能在不引入大规模架构改造的前提下，显著降低公网单实例部署时的被利用概率和运维误配风险。
